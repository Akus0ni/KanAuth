# 03 — EC2 (Elastic Compute Cloud)

## The Story: Renting A Kitchen By The Hour

Imagine you want to cook pizzas but you don't have a kitchen. You go to a
shared commercial kitchen that rents out **kitchen-in-a-box** by the hour.
You walk in, tell them "I want a small one with 2 burners and enough room
for 2 GB of ingredients", they hand you the keys, and you start cooking.

When you are done, you give the keys back and stop paying. If you want a
bigger kitchen tomorrow, you rent a bigger one. If one kitchen catches fire,
you rent another and move your recipe over.

**That is EC2.** EC2 (*Elastic Compute Cloud*) is AWS renting out virtual
computers by the second.

## What Is A Virtual Computer, Exactly?

AWS owns huge warehouses (*data centers*) full of physical servers — real
metal boxes with real CPUs, real RAM, and real disks. Each physical box is
carved up with **virtualization software** (a hypervisor) into many smaller
pretend-computers, each with their own pretend-CPU, pretend-RAM, and
pretend-disk.

From the inside, a pretend-computer feels exactly like a real one: it boots
Linux, you can SSH in, you can install software. It just happens to be
sharing the physical hardware with a few dozen other pretend-computers next
door. This is called a **virtual machine** or **VM** — AWS calls each one an
**instance**.

### What Does `t3.small` Mean?

AWS names instances like `<family><generation>.<size>`:

- **`t`** — "T-series" = burstable, cheap. Idle most of the time, bursts under load.
- **`3`** — third generation of that family.
- **`small`** — size: 2 vCPU, 2 GB RAM.

Other sizes on the T3 family:

| Size | vCPU | RAM | Good For |
|---|---|---|---|
| `t3.nano` | 2 | 0.5 GB | Tiny experiments |
| `t3.micro` | 2 | 1 GB | Free tier, static sites |
| `t3.small` | 2 | 2 GB | **KanAuth** — small APIs |
| `t3.medium` | 2 | 4 GB | Slightly bigger APIs |
| `t3.large` | 2 | 8 GB | |

> **"Burstable" means:** the CPU is limited to a baseline (e.g., 20%) but
> can temporarily run at 100% by spending "CPU credits" earned while idle.
> For a low-traffic API like KanAuth, this is perfect and cheap.

## What's Running On The EC2 Instance?

The EC2 is not a blank Linux box. EB bakes in several things at boot:

```
┌──────────────────────────────────────────────────────────────┐
│  EC2 instance (Amazon Linux 2)                               │
│                                                              │
│   nginx          ──► listens on port 80, proxies to Docker  │
│   ↓                                                          │
│   Docker daemon  ──► pulls image from ECR                   │
│   ↓                                                          │
│   KanAuth.API container                                      │
│     ↑                                                        │
│     env.list (from SSM via .ebextensions)                    │
│                                                              │
│   aws-cli       ──► used by .ebextensions to fetch secrets  │
│   CloudWatch    ──► agent streams logs & metrics out        │
│   eb-engine     ──► the EB platform software that runs the  │
│                     deploy lifecycle, health checks, etc.   │
└──────────────────────────────────────────────────────────────┘
```

Nginx is in front of Docker for two reasons:
1. It terminates raw HTTP cleanly.
2. EB's health reporting reads nginx access logs to compute a request rate.

## How Does A Request Reach The EC2?

```
Browser ──► DNS ──► ALB (public IP)
                      │
                      │ TCP 80 (inside VPC)
                      ▼
                   EC2 instance (private IP)
                      │
                      │ nginx upstream
                      ▼
                   Docker container port 80
                      │
                      │ Kestrel (ASP.NET Core)
                      ▼
                   AuthController
```

The EC2 has a **private IP** (e.g., `10.0.1.42`). It has no public IP
directly — the ALB is the only thing on the public internet. That means
even if someone knew the private IP, they could not reach it from outside AWS.

## Exact Steps: You Never Manually Create This EC2

With EB, you **do not run `aws ec2 run-instances`**. EB does it for you when
you run `eb create`. What you *can* do:

### Inspect the instance

```bash
aws ec2 describe-instances \
  --region ap-south-2 \
  --filters "Name=tag:elasticbeanstalk:environment-name,Values=kanauth-prod"
```

Shows you the instance ID, private IP, AMI, state, etc.

### SSH in (optional, for debugging)

```bash
eb ssh
```

**What it does:** Uses the SSH key pair you configured during `eb init` to
SSH into the EC2 instance.

**Why you'd do it:** Only for debugging. Tail logs, inspect disk, run
`docker ps`. In normal operation you never SSH in — everything important is
streamed to CloudWatch.

### Terminate and recreate (disaster recovery)

```bash
eb terminate kanauth-prod      # nukes EVERYTHING
eb create kanauth-prod ...     # creates it again
```

**When to use it:** As a nuclear option when state drifts so badly nothing
else works. All traffic is dropped for 5–10 minutes during rebuild.

## Auto Scaling: The Part EB Sets Up Silently

EB creates an **Auto Scaling Group (ASG)** even if you only have one
instance. The ASG says: "Keep exactly 1 healthy instance of type `t3.small`
running at all times." If the instance crashes, the ASG spins up a new one
within a few minutes. This is the **self-healing** property.

If you ever need more capacity, you edit the ASG to have min=2, max=10, and
EB will launch more instances automatically when CPU crosses a threshold.
The ALB spreads traffic across all healthy instances.

## Alternatives: Other Ways To Get Compute

| Alternative | What It Is | Why You'd Pick It Over EC2 |
|---|---|---|
| **AWS Fargate** | Serverless container runtime — AWS owns the underlying VM | You don't want to patch, size, or monitor VMs. Pay per vCPU-second. |
| **AWS Lambda** | Function-as-a-service — AWS runs your code on demand | Sporadic traffic, zero idle cost, 15-min max per request |
| **AWS Lightsail** | Dumbed-down EC2 with flat monthly pricing | Predictable budget, hobby projects |
| **Spot instances** | Normal EC2, up to 90% cheaper, but can be killed with 2-min notice | Batch jobs, fault-tolerant workloads |
| **Dedicated hosts** | An entire physical server, just for you | Licensing requirements (e.g., Oracle) |
| **EC2 bare metal** | Raw hardware, no hypervisor | Nested virtualization, real-time workloads |
| **AWS Outposts** | AWS hardware installed in your own data center | Data sovereignty / ultra-low latency on-prem |
| **Google Cloud Compute / Azure VMs** | The same concept on other clouds | Multi-cloud, existing commitments |

### When Is Plain EC2 The Wrong Choice?

- **You don't want to patch the OS.** → Go Fargate.
- **You want to pay $0 when nobody is using the app.** → Go Lambda.
- **You want to run a full Kubernetes cluster.** → Go EKS managed nodes.

KanAuth uses **EC2 via EB** because it is the simplest way to get a stable,
always-on box that can open a TCP connection to a private RDS.

## Key Takeaways

- **EC2 is a rented virtual computer.** `t3.small` is cheap, burstable, and enough for KanAuth.
- **You never create it manually** — EB does it for you inside an Auto Scaling Group.
- **nginx sits in front of Docker** on the instance to handle the ALB traffic cleanly.
- **The instance has no public IP** — only the ALB talks to the public internet.
- **Alternatives include Fargate, Lambda, Lightsail, EKS.** Each moves you up or down the "less work ↔ more control" spectrum.
